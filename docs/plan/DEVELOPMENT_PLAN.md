# 开发安排建议

## 整体推进顺序
1. 先完成环境准备和工程启动
2. 再统一表结构、字段名、接口入参与返回格式
3. 再按服务边界并行开发
4. 然后按链路顺序联调
5. 最后补压测、异常处理、文档与提交材料

## 第 1 阶段：所有人先完成的准备工作
- 拉取项目并确认 `mvn clean package -DskipTests` 可以通过
- 导入 `sql/schema.sql`
- 阅读 `README.md`、`docs/ARCHITECTURE.md`、`team/DIVISION.md`
- 确认自己负责的服务、数据库表、上下游调用关系
- 统一好接口字段命名、时间格式、金额精度、错误码格式

## 第 2 阶段：五个成员分别先做什么

### 成员 A：观众服务上手步骤
1. 阅读 `viewer-service` 和 `donation-common`
2. 先定义打赏接口 DTO、返回体、Controller 路径
3. 补 Service 层，拆出打赏、画像查询、Top10 查询三个核心流程
4. 预留 Feign 客户端，定义对财务服务和经营分析服务的调用接口
5. 先用假数据把接口跑通，再切到真实下游服务
6. 最后补超时、熔断、降级逻辑

### 成员 B：财务服务上手步骤
1. 阅读 `finance-service`、`sql/schema.sql`
2. 先确定入账表、提成规则表、余额表对应的实体和 Mapper
3. 实现打赏入账接口，先保证一条打赏只入账一次
4. 实现提成配置接口，明确生效时间字段逻辑
5. 实现余额查询接口，保证旧比例和新比例按生效时间分别计算
6. 最后补事务、一致性校验和重复请求保护

### 成员 C：经营分析服务上手步骤
1. 阅读 `analytics-service`、画像表和统计表结构
2. 先完成观众画像查询接口返回结构
3. 设计画像计算逻辑，按消费金额排序划分高、中、低消费人群
4. 实现按小时、主播、性别组合查询接口
5. 实现定时任务，定时从打赏明细表汇总到统计表和画像表
6. 联调前准备好可复核的样例数据

### 成员 D：模拟服务上手步骤
1. 阅读 `simulator-service` 和观众服务打赏接口定义
2. 实现模拟请求参数结构，如主播数、观众数、QPS、持续时间
3. 实现调用观众服务的压测逻辑
4. 实现异常场景模拟，如超时、部分请求失败、重试观测
5. 输出一份压测使用说明，说明如何压到每节点每秒 500 请求
6. 联调时负责构造稳定可重复的测试数据

### 成员 E：公共支撑成员上手步骤
1. 审核并维护 `sql/schema.sql`
2. 维护 `donation-common` 中的公共返回体、traceId、异常结构
3. 统一 `application-local.yml` 的数据库、端口、注册中心配置
4. 检查所有服务的启动方式、日志格式、包结构是否统一
5. 组织联调顺序，记录问题并分发修复任务
6. 维护项目说明、分工说明、环境说明和最终提交材料

## 第 3 阶段：联调步骤建议
- 第一次联调：只联通“模拟服务 -> 观众服务 -> 财务服务”
- 第二次联调：联通“观众服务 -> 经营分析服务”
- 第三次联调：把四个服务一起启动，验证完整业务链路
- 第四次联调：压测、超时、熔断、降级、幂等和重复请求处理

## 第 4 阶段：验收前每个人必须完成的事
- 每个人负责的服务必须能独立启动
- 每个人负责的接口必须至少自测通过一次
- 每个人要补一份自己模块的接口说明或功能说明
- 所有人一起确认数据库字段、返回结构、错误提示一致
- 最终由成员 E 汇总提交材料
