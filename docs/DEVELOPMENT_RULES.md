# 开发细则与注意事项

## 代码规则
- 每个服务只负责自己的业务边界
- 公共类型优先放到 `donation-common`
- Controller 只做参数接收与结果返回
- Service 负责业务编排与跨服务调用
- Mapper 只做数据访问，不写业务逻辑
- 不要把一个服务的业务实现直接写进另一个服务

## 接口规则
- 统一返回 `ApiResponse`
- 所有接口都要带 `traceId`
- 打赏相关接口必须考虑幂等
- 所有外部调用必须设置超时
- 观众画像查询必须在超时后快速返回友好提示
- 统一使用 JSON 作为请求和响应格式
- 金额字段统一使用 `BigDecimal`
- 时间字段先统一为字符串或 `LocalDateTime`，全组保持一致

## 日志规则
- 记录请求参数、关键业务 ID、结果状态
- 使用 `traceId` 关联请求链路
- 错误日志必须包含服务名、接口名、入参摘要
- 不要把整段无关大对象直接打印到日志

## 五人协作注意事项
- 成员 A、B、C、D 在开发前必须先和成员 E 对齐字段名和接口契约
- 修改公共模块前先同步，避免把别人代码改挂
- 表结构一旦修改，要同步更新 SQL、实体类、Mapper 和文档
- 下游服务未完成前，可以先返回假数据，但要明确标记并尽快替换
- 联调问题按链路定位，不要一开始就全项目一起排查

## 推荐开发顺序
- 先完成 Controller、DTO、返回结构
- 再完成 Service 主流程
- 再补数据库访问和下游调用
- 最后补异常处理、超时、熔断、幂等和日志

## 每个人开发时要注意什么

### 成员 A：观众服务
- 重点不是只把接口写出来，而是要把跨服务调用链路设计清楚
- 画像查询必须有超时控制和友好提示
- Top10 查询优先调用经营分析结果或汇总表，不要直接全表扫描明细表

### 成员 B：财务服务
- 入账一定要防重复，优先用唯一键配合业务幂等
- 提成比例变更要按生效时间分段计算，不要直接覆盖历史数据
- 余额计算尽量走清晰的结算逻辑，不要把规则分散到多个类

### 成员 C：经营分析服务
- 画像逻辑要先和组员讲清楚口径，避免联调时发现理解不一致
- 定时统计允许延迟，但要保证结果可复现
- 多维查询返回前要想清楚时间范围边界和分组维度格式

### 成员 D：模拟服务
- 不要一开始就盲目压大流量，先用小流量验证接口正确性
- 请求失败时要记录失败原因，方便联调定位问题
- 压测参数要做成可配置，不要把 QPS 写死

### 成员 E：公共支撑
- 不要只写文档，要真正负责拉齐字段、配置、日志、启动方式
- 每次联调后都记录问题清单和责任人
- 最终提交前统一检查目录结构、SQL、说明文档和运行步骤

## 新手上手建议
- 第一步先把自己负责的服务单独启动起来
- 第二步先打通一个最简单接口
- 第三步再接数据库或下游服务
- 第四步完成主流程后再处理异常和优化项
- 遇到问题先看启动日志、请求参数、数据库数据，再判断是谁的问题
